# UX Lot 1 — décisions et implémentation (2026-07-19)

> Suite à un audit UX externe du parcours opérateur. Chaque constat a
> été vérifié contre le code avant décision ; ce document est le
> registre de ce qui a été **adopté**, **planifié** et **rejeté**, pour
> que les futures sessions ne re-litiguent pas les mêmes points.

## Vérification préalable — deux corrections de tir

1. **« La feuille ne pilote pas le G-code »** : à moitié vrai
   seulement. Le placement envoyé à `/generate` est la bbox réelle des
   dessins posés, en coordonnées machine absolues (transform identité)
   — le WYSIWYG plan → papier est réel. Ce qui est vrai : la *zone
   papier* est une aide au placement, pas une frontière d'impression.
   Décision : renommage honnête (« Repère papier » + hint), **pas** de
   refactor « feuille canonique » (grosse casse de golden tests G-code
   pour un gain que l'opérateur ne voit pas).
2. **« Onglet Fichiers ambigu »** : le libellé réel est déjà
   « Fichiers G-code », la collision est moindre que décrit. Renommage
   éventuel groupé avec une future passe de vocabulaire.

## Planifié (2e vague, non commencé)

- **`PrimaryWorkflowAction`** — action principale contextuelle unique
  (machine à états à centraliser depuis `canPlay` / `canGenerate` /
  `preflight` / `queue.active`). Contrainte posée : ne pas retirer
  Pause/Stop du header pendant un run (distance de pouce fixe).
- **« Détecter et connecter »** — demande un endpoint backend
  d'énumération de ports (pyserial `list_ports`, déjà en dépendance)
  + heuristique d'essai. Seul point de l'audit qui touche le backend.
- **Passe de vocabulaire groupée** — Simple/Avancé, Tracés
  enregistrés, Appliquer : à faire d'un bloc (les renommages éparpillés
  cassent les assertions i18n au fil de l'eau).

## Rejeté / gelé (avec motif)

- **Feuille canonique (Lot 3.1-3.3 de l'audit)** — cf. §Vérification :
  le modèle actuel est déjà WYSIWYG ; le refactor changerait
  `_make_transform`, le homing et le préflight pour un gain conceptuel.
- **Suppression de la visite guidée** — one-shot, discrète, testée ;
  la remplacer par des hints contextuels coûte plus qu'elle ne rapporte.
- **Refonte de navigation « Plan → Aperçu → Imprimer »** — gelée
  jusqu'au bilan de l'action contextuelle unique (elle en dépend).

## Vague 2 — livrée (2026-07-19, même branche)

Les trois chantiers « planifiés » ci-dessus ont été implémentés :

1. **Détecter et connecter** — `GET /plotter/ports` (pyserial, candidats
   USB-serial en premier, entrée synthétique sous fake hardware),
   `detectAndConnect()` dans le store (essaie les candidats dans
   l'ordre, adopte le premier qui répond), onglet connexion refondu
   (bouton primaire + « Configuration manuelle » repliée).
2. **Vocabulaire** — Simple/Avancé, « Tracés enregistrés », « Appliquer »
   (FR/EN, catalogues de tests alignés, fait d'un bloc comme décidé).
3. **PrimaryWorkflowAction** — `useWorkflowStep` centralise la machine à
   états (run actif > import > placement > style > stylos > génération >
   connexion > lancement) ; la barre sous le plan affiche UNE action
   primaire contextuelle (+ Arrêter en secondaire destructif pendant un
   run). Chaque action réutilise les chemins existants : picker de
   FilesPane (`ui.requestAddFile`), `createPlacementFromLibrary`,
   éditeur, `openPlotterSettings('colors')`, `generate()` + bascule
   simulation, `detectAndConnect()` avec repli formulaire, et
   `useLaunchCurrentJob` (gate magasin + odomètre inclus). Conformément
   à la décision, le transport du header est conservé pendant un run.
